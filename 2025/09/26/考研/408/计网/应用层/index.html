计算机网络：应用层

网络应用模型

  • 客户/服务器模型（Client/Server, C/S模型）

    • 定义：一种网络应用架构，存在一个始终开启的“服务器”主机，为多个“客户机”主机提供服务，客户机主动发起请求，服务器被动响应请求。
    • 工作流程：
      1. 服务器处于接收请求的等待状态。
      2. 客户机发出服务请求，并等待接收结果。
      3. 服务器收到请求后，分析请求、处理请求，将结果发送给客户机。
    • 结构/组成：
      • 服务器：运行专门服务程序，需提前开启，被动等待请求，无需知道客户机地址。
      • 客户机：运行客户程序，需知道服务器地址（如IP/域名），主动发起请求。
    • 性质/特点：
      • 网络中计算机地位不平等：服务器可通过权限限制管理客户机，网络管理集中方便。
      • 客户机之间不直接通信：例如Web应用中，两个浏览器无法直接交互。
      • 可扩展性不佳：受服务器硬件性能和网络带宽限制，支持的客户机数量有限。
    • 典型应用场景：Web服务、文件传输（FTP）、远程登录、电子邮件等。
    • 优点：管理集中，服务稳定性高，便于权限控制。
    • 缺点：服务器易成为系统瓶颈，面对大量请求时性能下降明显。

  • P2P模型（Peer-to-Peer模型）

    • 定义：一种无固定客户机与服务器划分的网络应用架构，任意“对等方（Peer）”既可作为客户机访问其他节点资源，也可作为服务器提供资源，节点权利与义务大体对等。
    • 核心逻辑：本质仍隐含C/S逻辑（每个节点兼具两种角色），但无中心服务器，资源分散存储于各节点。
    • 性质/特点：
      • 节点地位对等：无固定服务器，每个节点均参与资源上传与下载。
      • 资源分布式存储：传输内容不依赖中心服务器，由节点间直接交换。
    • 优点/适用场景：
      • 减轻服务器计算压力：任务分配到各节点，提升系统效率与资源利用率。
      • 可扩展性好：无传统服务器的响应/带宽限制，支持大量节点接入。
      • 网络健壮性强：单个节点失效不影响其他节点正常通信，适用于文件共享（如P2P下载）。
    • 缺点/局限性：
      • 占用节点资源多：节点需同时提供服务，消耗内存、硬盘资源（如P2P下载损伤硬盘）。
      • 易导致网络拥塞：据统计，P2P程序曾占互联网50%~90%流量，多数ISP（互联网服务提供商）持反对态度。

域名系统（Domain Name System, DNS）

  • 定义：因特网的命名系统，用于将人类易记的主机名（如www.cskaoyan.com）转换为机器可处理的IP地址，采用客户/服务器（C/S）模型，运行于UDP之上，使用53号端口。

  • 核心组成：从概念上分为层次域名空间、域名服务器、解析器三部分。

  • 层次域名空间

    • 结构：采用层次树状结构，每个连接到因特网的主机/路由器拥有唯一“域名（Domain Name）”，域可划分子域，形成顶级域、二级域、三级域等层级。
    • 域名格式：
      • 由标号序列组成，标号间用“.”分隔（如www.cskaoyan.com）。
      • 标号规则：
        • 英文不区分大小写。
        • 仅允许包含连字符（-），无其他标点符号。
        • 单个标号最长63字符，完整域名最长255字符。
        • 级别最低的域名在最左，级别最高的顶级域名在最右。
    • 顶级域名（Top Level Domain, TLD）分类：
      • 国家（地区）顶级域名（nTLD）：如“cn”（中国）、“us”（美国）、“uk”（英国）。
      • 通用顶级域名（gTLD）：如“com”（公司）、“net”（网络服务机构）、“org”（非营利组织）、“edu”（教育机构）、“gov”（政府部门）。
      • 基础结构域名（arpa）：用于反向域名解析（IP地址→域名）。
    • 域名管理机制：各级域名由上一级机构管理，顶级域由ICANN（因特网名称与数字地-址分配机构）管理，国家顶级域下的二级域由各国自行管理（如中国将edu.cn委托给CERNET）。

  • 域名服务器

    • 定义：运行域名解析程序的服务器，负责管辖“区（Zone，小于等于域）”，保存区内主机的域名-IP映射，且具备访问其他域名服务器的能力。
    • 分类及作用：
      • 根域名服务器：
        • 定义：最高层次的域名服务器，全球共13个（实际为冗余集群）。
        • 作用：知晓所有顶级域名服务器的域名与IP，本地域名服务器无法解析时首先求助于它；不直接返回IP，仅告知下一级顶级域名服务器地址。
      • 顶级域名服务器：
        • 作用：管理所属顶级域下的所有二级域名，收到查询时返回最终IP或下一级权限域名服务器地址。
      • 权限域名服务器（授权域名服务器）：
        • 作用：每个主机需在此注册，确保能将管辖的主机名转换为IP；一台主机建议配置至少两个以保证可靠性，常同时充当本地域名服务器。
      • 本地域名服务器：
        • 定义：由ISP、大学、院系等提供，是域名解析的“入口”。
        • 作用：主机发起DNS查询时，请求首先发送至此；Windows配置“本地连接”时填写的DNS地址即本地域名服务器地址。

  • 域名解析过程

    • 定义：将域名转换为IP地址的过程，客户端通过DNS客户构造请求报文，以UDP数据报发送至本地域名服务器。
    • 查询方式：
      • 递归查询：
        • 定义：若本地域名服务器无法解析，以DNS客户身份代主机继续查询（如向根、顶级域名服务器请求），最终将结果返回主机。
        • 特点：主机仅发送1次请求，本地域名服务器负责全程查询；因给根服务器造成大负载，实际极少使用。
      • 迭代查询：
        • 定义：本地域名服务器向其他服务器查询时，对方仅返回“IP”或“下一级查询地址”，由本地域名服务器自行后续查询。
        • 特点：本地域名服务器需多次发送请求（如根→顶级→权限），是实际中本地域名服务器向根查询的默认方式。
    • 典型解析步骤（以查询y.abc.com为例）：
      1. 客户机向本地域名服务器发送递归查询请求。
      2. 本地域名服务器无缓存，以客户身份向根域名服务器发送迭代查询。
      3. 根域名服务器返回.com顶级域名服务器（dns.com）的IP。
      4. 本地域名服务器向dns.com发送迭代查询，后者返回abc.com权限域名服务器（dns.abc.com）的IP。
      5. 本地域名服务器向dns.abc.com发送迭代查询，后者返回y.abc.com的IP。
      6. 本地域名服务器缓存结果，并将IP返回给客户机。
    • 高速缓存机制：
      • 作用：域名服务器缓存近期查询的域名-IP映射，后续相同查询可直接返回结果，减少因特网DNS报文数量，提升解析效率。
      • 特性：映射非永久有效，服务器会定期丢弃缓存；主机也会维护本地缓存，仅在缓存未命中时请求域名服务器。

文件传输协议（File Transfer Protocol, FTP）

  • 定义：因特网上广泛使用的文件传输协议，提供交互式访问，允许客户指明文件的类型和格式，屏蔽异构系统细节，支持不同主机间文件传输、权限管理、匿名共享。

  • 核心功能：

    • 支持异构主机（硬/软件体系不同）间的文件传输。
    • 以用户权限管理方式实现对远程FTP服务器文件的管理。
    • 以匿名FTP方式提供公用文件共享。

  • FTP工作原理

    • 模型：采用C/S模型，使用TCP提供可靠传输服务，服务器进程由“主进程”和“从属进程”组成。
    • 服务器进程分工：
      • 主进程：打开21号控制端口，等待客户连接；收到连接请求后启动从属进程处理，自身返回等待状态（与从属进程并发执行）。
      • 从属进程：处理单个客户的请求，完成后终止。
    • 工作步骤：
      1. 服务器监听21号端口（控制端口），等待客户连接。
      2. 客户机发起连接请求，与服务器建立控制连接。
      3. 服务器启动从属进程，处理客户的文件传输请求。
      4. 完成传输后，从属进程终止，主进程继续等待新请求。
    • 关键特性：服务器需在会话期间保留用户状态（如用户账户、当前远程目录位置）。

  • 控制连接与数据连接

    • 核心逻辑：FTP工作时使用两个并行的TCP连接，分离控制信息与数据传输，简化协议实现。
    • 控制连接：
      • 端口：服务器端使用21号端口。
      • 作用：传输控制信息（如连接请求、传送命令、中止指令），不传输文件；整个会话期间保持打开。
      • 建立过程：服务器监听21端口，客户机发起连接，建立后用于全程控制交互。
    • 数据连接：
      • 端口：服务器端默认使用20号端口（主动模式）。
      • 作用：连接客户端与服务器端的数据传送进程，实际完成文件/文件列表的传输；传输完毕后关闭。
      • 传输模式：
        • 主动模式（PORT）（默认模式）：客户端连接服务器21端口，登录后开放随机端口并告知服务器；服务器通过20端口连接客户端随机端口，发送数据。
        • 被动模式（PASV）：客户端发送PASV命令，服务器开放随机端口并告知客户端；客户端连接该随机端口，接收数据。
        • 选择权：由客户端决定使用哪种模式。
    • 带外传送：因控制连接与数据连接分离，FTP的控制信息属于“带外（Out-of-band）”传送（区别于HTTP的“带内”传送）。
    • 与NFS（Network File System）的区别：FTP需来回传输完整文件修改，NFS允许直接读写远程文件片段，更高效。

电子邮件（E-mail）

  • 电子邮件系统的组成结构

    • 核心构件：由用户代理、邮件服务器、应用层协议（SMTP、POP3等）三部分组成。
    • 各构件功能：
      • 用户代理（User Agent, UA）：
        • 定义：用户与电子邮件系统的接口，通常为客户端软件（如Outlook、Foxmail）。
        • 功能：提供撰写、显示、处理邮件的界面，支持发送（调用SMTP）和接收（调用POP3/IMAP）邮件。
      • 邮件服务器：
        • 功能：发送/接收邮件，报告传送状态（已交付、被拒绝、丢失）；同时充当SMTP客户（发送邮件时）和服务器（接收邮件时）。
        • 关键作用：暂存邮件至收件人邮箱，等待收件人读取。
      • 协议分类：
        • 发送协议：SMTP（推模式），用于用户代理→邮件服务器、邮件服务器→邮件服务器的邮件推送。
        • 读取协议：POP3/IMAP（拉模式），用于用户代理从邮件服务器拉取邮件。
    • 典型收发流程：
      1. 发件人通过用户代理撰写邮件，点击“发送”。
      2. 用户代理通过SMTP将邮件传送给发送端邮件服务器。
      3. 发送端服务器将邮件放入缓存队列，等待发送。
      4. 发送端SMTP客户与接收端SMTP服务器建立TCP连接，发送邮件至接收端服务器。
      5. 接收端服务器将邮件存入收件人邮箱。
      6. 收件人通过用户代理，使用POP3/IMAP从接收端服务器取回邮件。

  • 电子邮件格式与MIME

    • 电子邮件格式：
      • 组成：分为“信封”和“内容”，内容又分“首部”和“主体”；信封信息由系统自动从内容首部提取。
      • 内容首部：
        • 必需字段：To（收件人邮箱地址，格式为“用户名@主机域名”，如abc@cskaoyan.com）、From（发件人地址，系统自动填入）。
        • 可选字段：Subject（邮件主题）。
        • 分隔符：首部与主体之间用空行分隔。
      • 内容主体：用户自由撰写的邮件正文。
    • 多用途因特网邮件扩展（Multipurpose Internet Mail Extensions, MIME）
      • 背景：SMTP仅支持传输7位ASCII码文本，无法传输非英语文字（如中文）、二进制文件（如图片、可执行文件），故引入MIME。
      • 核心作用：不改动/取代SMTP，仅将非ASCII数据转换为ASCII格式，适配SMTP传输；接收端再逆转换恢复原始数据。
      • 组成内容：
        • 5个新邮件首部字段（如MIME版本、内容类型、传送编码）。
        • 标准化多媒体内容格式（文字、图像、语音等）。
        • 定义传送编码规则，确保数据不被邮件系统修改。

  • SMTP与POP3

    • SMTP（Simple Mail Transfer Protocol, 简单邮件传输协议）：
      • 定义：提供可靠电子邮件传输的应用层协议，采用C/S模型，运行于TCP之上，使用25号端口。
      • 角色划分：发送邮件的进程为SMTP客户，接收邮件的进程为SMTP服务器。
      • 通信三阶段：
        1. 连接建立：发送端服务器SMTP客户扫描邮件缓存，发现邮件后与接收端服务器SMTP服务器建立TCP连接；接收端返回“220 Service ready”，客户发送HELO命令（附主机名）。
        2. 邮件传送：
          • 发送MAIL命令（附发件人地址），服务器返回“250 OK”。
          • 发送1个或多个RCPT命令（附收件人地址），验证收件人有效性。
          • 发送DATA命令，服务器返回“354 Start mail input”；客户传输邮件内容，以<CRLF>.<CRLF><CRLF>.<CRLF><CRLF>.<CRLF>结束。
        3. 连接释放：客户发送QUIT命令，服务器返回“221 Service closing”，释放TCP连接。
      • 关键特性：不使用中间邮件服务器，TCP连接直接在发送端与接收端服务器之间建立；连接失败时，发送端服务器定期重试。
    • POP3（Post Office Protocol 3, 邮局协议版本3）：
      • 定义：简单的邮件读取协议，采用C/S模型，运行于TCP之上，使用110号端口。
      • 功能：支持用户代理从邮件服务器读取邮件，仅提供基础读取能力。
      • 工作方式：
        • 下载并保留：读取后邮件仍保存在服务器，可再次读取。
        • 下载并删除：读取后邮件从服务器删除。
      • 局限性：功能简单，无文件夹管理、部分读取等高级功能。
    • IMAP（Internet Message Access Protocol, 因特网报文存取协议）：
      • 特点：比POP3复杂，支持创建文件夹、移动邮件、远程查询、部分读取（如仅读首部）。
      • 优势：维护用户会话状态，适配低带宽场景（如仅下载大邮件的部分内容）。
    • 基于Web的电子邮件：如Hotmail、Gmail，用户浏览器与邮件服务器间使用HTTP，服务器间仍使用SMTP。

万维网（World Wide Web, WWW）

  • WWW的概念与组成结构

    • 定义：分布式、联机式的信息存储空间，通过超链接按需求获取资源，由URL、HTTP、HTML三大标准支撑。
    • 核心标准：
      • 统一资源定位符（Uniform Resource Locator, URL）：
        • 作用：唯一标识万维网资源的位置与访问方法。
        • 格式：<协议>://<主机>:<端口>/<路径>（如http://www.tsinghua.edu.cn/chn/index.htm）；协议（如http、ftp）、主机（域名/IP）为必需，端口、路径可省略；不区分大小写。
      • 超文本传输协议（HyperText Transfer Protocol, HTTP）：
        • 作用：定义浏览器与Web服务器的请求/响应格式与规则，基于TCP提供可靠传输；是万维网文件交换的基础。
      • 超文本标记语言（HyperText Markup Language, HTML）：
        • 作用：描述网页结构与内容格式（文字、图像、视频等），支持超链接跳转。
    • 工作模型：采用C/S模型，浏览器为客户程序，存储Web文档的主机为Web服务器。
    • 典型工作流程：
      1. 浏览器解析用户输入的URL，发起连接请求。
      2. Web服务器将URL转换为文件路径，生成响应信息。
      3. 传输响应后，关闭TCP连接，浏览器解析并显示网页。

  • 超文本传输协议（HTTP）

    • 定义：面向事务（Transaction-oriented）的应用层协议，规定浏览器与服务器间的交互规则，支持可靠文件交换。
    • 操作过程（以访问www.tsinghua.edu.cn为例）：
      1. 浏览器分析URL（http://www.tsinghua.edu.cn/chn/index.htm）。
      2. 浏览器向DNS请求解析www.tsinghua.edu.cn的IP地址。
      3. DNS返回服务器IP，浏览器与该IP建立TCP连接（默认端口80）。
      4. 浏览器发送HTTP请求（如GET /chn/index.htm）。
      5. 服务器通过HTTP响应发送index.htm文件。
      6. 释放TCP连接，浏览器解析并显示网页。
    • 核心特点：
      • 基于TCP：依赖TCP保证可靠传输，无需处理重传等细节。
      • 无连接：虽使用TCP连接，但通信双方交换HTTP报文前无需建立HTTP连接。
      • 无状态：服务器不记忆客户历史请求，简化服务器设计；通过Cookie+数据库跟踪用户状态（如浏览记录、登录状态）。
        • Cookie工作原理：服务器为用户生成唯一识别码，存入响应的Set-cookie首部；用户下次请求时携带Cookie首部，服务器通过识别码查询用户状态。
      • 连接方式：
        • 非持续连接（HTTP/1.0）：
          • 每个网页元素（如文本、图像）的传输需单独建立TCP连接；
          • 请求一个元素需2×RTT（1个RTT建立TCP，1个RTT请求/接收），
          • 服务器负担重；
          • 浏览器常通过多并行TCP连接提速。
        • 持续连接（HTTP/1.1默认）：TCP连接在发送响应后保持打开，支持后续请求复用。
          • 非流水线方式：客户收到前一个响应后才发送下一个请求，TCP连接存在空闲。
          • 流水线方式：客户连续发送多个请求，服务器连续响应，仅需1×RTT延迟，效率更高。
    • HTTP报文结构：
      • 分类：分为请求报文（客户→服务器）和响应报文（服务器→客户），均由“开始行+首部行+实体主体”组成。
      • 请求报文：
        • 开始行（请求行）：格式为“方法 URL HTTP版本”（如GET /index.htm HTTP/1.1）。
        • 常用方法：
          • GET：请求读取URL标识的资源。
          • HEAD：请求读取URL标识资源的首部（无主体）。
          • POST：向服务器提交信息（如表单数据）。
          • PUT：在URL下存储文档。
          • DELETE：删除URL标识的资源。
        • 首部行：包含
          • Host（服务器域名）、
          • Connection（连接方式）、
            • Keep-Alive
            • Close
          • User-Agent（浏览器信息）等字段。
        • 实体主体：通常为空（GET请求）或包含提交数据（POST请求）。
      • 响应报文：
        • 开始行（状态行）：格式为“HTTP版本 状态码 短语”（如HTTP/1.1 200 OK）。
        • 常见状态码：
          • 202 Accepted：请求已接受。
          • 400 Bad Request：错误请求。
          • 404 Not Found：请求资源不存在。
        • 首部行：包含服务器信息、内容类型、Cookie设置等。
        • 实体主体：通常为请求的网页/文件内容。

动态主机配置协议（Dynamic Host Configuration Protocol, DHCP）

  • 定位：应用层协议（运行在 UDP 之上，服务器端口 67，客户端端口 68），用于在主机加入网络时自动获取网络参数（IP 地址、子网掩码、默认网关、DNS 服务器等）。
  • 引入动机：手工配置易出错、难集中管理，且地址需要重复利用（移动 / 临时接入主机）。
  • 工作思想：采用“租约（Lease）”机制，地址动态分配，过期归还地址池，提高地址利用率。
  • 基本报文类型（核心考点）：
    1. DHCP Discover（客户端广播寻找服务器）
    2. DHCP Offer（服务器提供可用地址与参数）
    3. DHCP Request（客户端选择并请求确认该地址）
    4. DHCP ACK（服务器确认租约并下发最终参数）
      （整体常记为 DORA 过程）
  • 交互特点：前 3 个阶段多使用广播（因为客户端尚无 IP / 不知服务器地址），最终确认后客户端使用获配地址正常通信。
  • 租约续约（可理解，不是重点）：客户端租期过半尝试直接向原服务器单播 Request 续租，若失败再广播（若涉及 T1/T2 定时器等细节 → 标记：超纲）。
  • 与 BOOTP 的关系（了解）：DHCP 向后兼容 BOOTP，增加“租约 + 扩展选项”能力（属于实现背景，超纲）。
  • 常见考查点速记：
    • 属于“即插即用”配置协议，用于获得 IP 与其他网络参数。
    • 使用 UDP（67/68），不用 TCP 的原因：启动阶段无可靠端到端路径、实现简化。
    • 采用租约而不是永久分配，以便地址复用。

边界网关协议（Border Gateway Protocol, BGP）

  • 定位：自治系统（AS）之间的“域间”路由选择协议（网络层路由控制范畴），本质使用 TCP 端口 179 作为其传输层承载。（放在此处为补充延伸，严格按层次它讨论的是“路由选择”，非典型应用层 408 应用协议列表，核心原理在 408 路由章节出现；本段仅概括性总结。）
  • 产生背景：因特网规模扩大，需支持跨自治系统的策略控制与可扩展性；传统距离向量 / 链路状态在域间不适用（策略、规模、自治需求）。
  • 协议类别：路径向量（Path Vector）协议——在“距离”之外，通告一条路由所经过的 AS 序列，防环并支持策略过滤。
  • 基本要素（考纲相关概念）：
    • 邻居（Peer）/ 会话：相邻 AS 的边界路由器之间建立 TCP 会话（保持心跳/Keepalive）。
    • 路由信息单位：前缀（IP 前缀 + 属性）。
    • 关键属性（核心需理解用途）：
      • AS-PATH：经由的 AS 列表，用于环路检测与策略（越短通常越优）。
      • NEXT-HOP：到达该前缀应采用的下一跳路由器 IP。
      • LOCAL-PREF（本地优先级，常在同一 AS 内决策出口）。（深入策略细节可视为超纲）
      • MED（多出口判定辅助值）。（较深入，超纲）
  • 报文类型（了解即可）：OPEN、UPDATE、KEEPALIVE、NOTIFICATION（报错/断开）。（详尽字段解析属超纲）
  • 与 OSPF / RIP 区分要点：
    • 作用范围：BGP 跨 AS，RIP/OSPF 用于 AS 内。
    • 关注点：BGP 强调“策略 + 可扩展性”，OSPF 强调“全网拓扑统一视图（链路状态）”，RIP 简单距离向量（最大跳数 15）。
    • 协议类型：BGP = 路径向量（Path + Policy）。
  • 常见考试粒度：识别 BGP 的作用（域间路由 / 使用路径向量 / 通告 AS-PATH）、运行在 TCP 之上（端口 179）。更细策略与属性优化（如 Communities, Route Reflector, Confederation）→ 超纲。

注：本节 BGP 扩展部分已用“超纲”标注；复习 408 重点请聚焦“BGP 解决域间路由，使用路径向量 + AS-PATH，基于 TCP 179 端口”。

本章小结及疑难点

  • 客户进程端口号与服务器进程端口号

    • 核心区别：
      • 服务器进程端口号：为应用层协议默认的“熟知端口号”（如HTTP 80、SMTP 25），固定不变，供客户进程识别连接目标。
      • 客户进程端口号：由客户端系统随机分配的“临时端口号”，无需提前固定；客户发起连接时，告知服务器自己的临时端口号，供服务器返回响应。
    • 连接建立逻辑：客户通过服务器的熟知端口号发起连接，服务器通过客户的临时端口号返回数据，形成双向通信。

  • 因特网（Internet）与万维网（WWW）的区别

    • 因特网：
      • 定义：基于TCP/IP族的全球性互连网络，是物理与逻辑结合的网络基础设施。
      • 组成：包含万维网、电子邮件、Usenet、新闻组等多种服务。
    • 万维网：
      • 定义：构建在因特网上的分布式信息系统，是因特网最主要的服务之一。
      • 核心：通过URL、HTTP、HTML实现资源访问与超链接跳转，仅覆盖因特网的部分功能。
    • 关系：万维网是因特网的子集，因特网为万维网提供底层通信支持。
