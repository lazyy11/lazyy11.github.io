计算机网络：传输层

传输层基础概念

  • 逻辑通信

    • 定义：对等层之间的通信看似沿水平方向传送，但两个对等层之间并无实际水平物理连接，是传输层为应用层进程提供的抽象通信服务。
    • 核心作用：屏蔽网络层以下的物理和数据链路层差异，实现端到端（进程间）的抽象通信。

  • 软件端口

    • 定义：应用层的各种协议进程与传输实体进行层间交互的地址，是传输层识别主机内不同应用进程的标识。
    • 性质/特点：
      • 与路由器/交换机的硬件端口完全不同，硬件端口是硬件设备交互接口，软件端口是逻辑地址。
      • 仅具有本地意义，即同一端口号在不同计算机上无关联。
    • 核心用途：传输层通过软件端口实现应用进程的“复用”与“分用”（多应用进程共享传输层服务）。
    • 复用与分用机制：
      • 复用：发送端多个应用进程（如浏览器、FTP客户端）可通过同一传输层协议（TCP/UDP）发送数据，传输层将不同进程的数据封装成对应协议的报文段（或数据报），通过端口号区分进程，实现“多进程共享传输层”。
      • 分用：接收端传输层收到报文后，依据“目的端口号+传输层协议类型”，将数据准确分发给对应的应用进程，实现“单一传输层服务分发至多进程”。

  • 端口号分类

    • 服务器端端口号
      • 熟知端口号：
        • 范围：0~1023
        • 性质：由IANA（互联网地址指派机构）分配给TCP/IP核心应用程序，全局统一。
        • 常用示例：
          • FTP：21
          • TELNET：23
          • SMTP：25
          • DNS：53
          • TFTP：69
          • HTTP：80
          • SNMP：161
      • 登记端口号：
        • 范围：1024~49151
        • 性质：供无熟知端口号的应用程序使用，需在IANA登记以避免重复。
    • 客户端端口号
      • 范围：49152~65535
      • 性质：又称“短暂端口号”，仅在客户进程运行时动态分配，通信结束后释放，可重复使用。
    • 长度：16比特，共216=655362^{16}=65536216=65536个不同端口号。

  • 套接字（Socket）

    • 定义：由IP地址与端口号拼接而成的通信端点，用于唯一标识网络中某台主机上的某个应用进程。
    • 结构：套接字(Socket) = (IP地址 : 端口号)
    • 核心作用：
      • 网络通信中，通过发送方和接收方的套接字识别通信端点。
      • 构成“返回地址”：源IP地址+源端口号，供接收方回传数据时定位发送方进程。

  • TCP连接的唯一标识（五元组）

    • 定义：用于在互联网中唯一区分一条TCP连接的五个关键信息组合，是传输层实现“分用”和连接管理的基础。
    • 组成：源IP地址、源端口号、目的IP地址、目的端口号、传输层协议（固定为TCP）。
    • 核心作用：即使两台主机间存在多个TCP连接（如同一客户端同时访问服务器的HTTP和FTP服务），也可通过五元组精准区分，避免数据分发错误。

UDP协议（用户数据报协议）

  • UDP协议特点

    • 性质/特点：
      • 无连接：通信前无需建立连接，直接发送数据。
      • 不可靠：接收方收到UDP数据报后无需向发送方发送确认，可能丢失、乱序。
      • 面向报文：不合并、不拆分应用层数据，保留报文边界，一次交付一个完整报文。
      • 首部简单：仅8字节，开销小，执行速度快，实时性好。
      • 仅提供基础服务：在IP层之上仅增加“多路复用/分用”和“数据错误检查”。
    • 与IP数据报的区别：IP数据报在网络层需经路由器存储转发，路由器可见IP首部；UDP数据报封装为IP数据报传输时，标准路由转发仅依赖IP首部，UDP首部字段一般不参与基本转发决策。

  • UDP首部结构

    • 总长度：8字节（固定，无选项字段）
    • 结构/组成（4个字段，每个字段2字节）：
      • 源端口号：发送方应用进程的端口号（可选，若为0表示无需回传）。
      • 目的端口号：接收方应用进程的端口号（必需，用于分用）。
      • UDP长度：UDP数据报的总长度（首部+数据，最小为8字节）。
      • UDP检验和：用于检查UDP首部和数据部分的传输错误（可选，但建议使用）。
    • 伪首部：
      • 定义：计算检验和时临时添加在UDP数据报前的12字节字段，非UDP真正首部，不传输、不递交。
      • 作用：将UDP数据报与IP地址关联，确保数据报交付到正确主机的正确进程。
      • 结构：源IP地址（4字节）、目的IP地址（4字节）、保留字段（1字节，0）、协议字段（1字节，UDP为17）、UDP长度（2字节）。

  • UDP检验和计算

    • 算法步骤：
      1. 发送方：将检验和字段置为全0，拼接伪首部与UDP数据报，形成临时数据报。
      2. 若临时数据报长度为奇数字节，在末尾补1个全0字节（不发送），将其拆分为多个16位字。
      3. 按二进制反码计算所有16位字的和，再对和取二进制反码，写入检验和字段并发送。
      4. 接收方：重复步骤1~2，计算临时数据报的16位字反码和，若结果为全1则无错，否则丢弃或报错（可以交付给上层，但必须附上错误报告）。
    • 与IP检验和的区别：IP检验和仅检验IP首部，UDP检验和检验伪首部、UDP首部及数据部分。

  • UDP适用场景

    • 优点/适用场景：
      • 实时性要求高的场景：如实时传输协议（RTP，用于音视频）。
      • 数据量小、对可靠性要求低的场景：如域名解析（DNS）、简单网络管理协议（SNMP）、简单文件传送协议（TFTP）。
      • 减少协议开销：适合对传输效率要求高于可靠性的场景。
    • 缺点/局限性：
      • 无可靠性保障，需应用层自行处理丢失、乱序问题。
      • 无流量控制，可能导致接收方缓冲区溢出。

TCP协议（传输控制协议）

  • TCP协议特点

    • 性质/特点：
      • 面向连接：通信前需通过“三次握手”建立连接，通信后需“四次挥手”释放连接。
      • 可靠传输：通过序号、确认、超时重传、流量控制、拥塞控制等机制确保数据不丢失、不重复、按序交付（段按顺序交付给应用层，但发送方到接收方的段顺序可能和序号不同）。
      • 面向字节流：将应用层数据视为连续字节流，拆分为TCP报文段传输，接收方重组。
      • 全双工通信：通信双方可同时发送和接收数据。
      • 无广播/多播：仅支持点对点通信。
      • 开销较大：首部至少20字节，需维护连接状态、执行复杂机制，占用更多资源。

  • TCP首部结构

    • 总长度：20 ~ 60字节（固定首部20字节，选项字段0 ~ 40字节，填充字段确保首部长度为4字节的整数倍）。
    • 结构/组成（核心字段）：
      • 源端口号（2字节）：发送方应用进程端口号。
      • 目的端口号（2字节）：接收方应用进程端口号。
      • 序号（4字节）：本报文段数据部分第一个字节的序号（范围0∼232−10 \sim 2^{32}-10∼232−1，可绕回）。
      • 确认号（4字节）：期望收到对方下一个报文段的第一个字节序号（仅当ACK=1时有效）。
      • 数据偏移（4位）：TCP首部长度（单位为4字节），最大值为15（对应60字节首部）。
      • 保留（6位）：预留未来使用，需置为0（现代TCP实现中，部分保留位可能用于扩展标志，如NS标志，RFC 3540）。
      • 标志位（6位，1位/标志，考研核心考经典6位，现代有扩展）：
        • URG（紧急）：URG=1时，紧急指针有效，优先传输紧急数据。
        • ACK（确认）：ACK=1时，确认号有效（连接建立后所有报文段需置1）。
        • PSH（推送）：PSH=1时，接收方需立即将数据交付应用层，不等待缓冲区满。
        • RST（复位）：RST=1时，表示连接出现严重差错，需释放连接并重新建立。
        • SYN（同步）：SYN=1时，表示连接请求或接受报文（SYN报文段不携带数据，但消耗序号）。
        • FIN（终止）：FIN=1时，表示发送方数据已发送完毕，请求释放连接（消耗序号）。
        • 现代扩展备注：现代TCP还支持CWR（拥塞窗口减少，响应ECN）、ECE（显式拥塞通知，指示网络拥塞），但考研答题以经典6位标志位为主。
      • 窗口（2字节）：接收方当前的接收窗口大小（单位为字节），用于流量控制，告知发送方可发送的最大数据量。
      • 检验和（2字节）：检验范围包括TCP首部、数据部分及伪首部（结构与UDP伪首部类似，协议字段为6）。
      • 紧急指针（2字节）：URG=1时，指示紧急数据在报文段中的结束位置（相对于序号的偏移量）。
      • 选项字段（可变）：常见选项包括最大报文段长度（MSS，Maximum Segment Size，三次握手协商，避免IP分片）、窗口扩大因子、时间戳（用于精确计算RTT、防止序号绕回的PAWS机制）等。

TCP差错检测与检验和

  • 检验和字段作用

    • TCP 首部中的 16 位“检验和”字段用于检测报文段（首部 + 数据）在传输过程中是否发生比特错误。
    • 检验范围包括 TCP 伪首部、TCP 首部、TCP 数据，与 UDP 的检验范围一致，但 对 IPv4/IPv6 都是强制字段（发送方必须计算、接收方必须验证）。

  • 伪首部结构（12 字节）

    • 源 IP 地址（4B）
    • 目的 IP 地址（4B）
    • 保留字段（1B，取 0）
    • 协议字段（1B，TCP 为 6）
    • TCP 长度（2B，首部 + 数据总长度）
    • 伪首部仅在计算检验和时临时拼接，不随报文段发送，用于把端到端身份信息纳入校验，防止“串线”。

  • 计算步骤（与 UDP 相同）

    1. 发送方：将检验和字段置 0，把伪首部 + TCP 首部 + TCP 数据串接成 16 位字序列（若字节数为奇数，临时补 1 字节 0）。
    2. 对所有 16 位字执行二进制反码加法，得出结果后再取一次反码，写入检验和字段。
    3. 接收方：按照同样方法求和（包含接收到的检验和），若结果全为 1，则认为比特级无错；否则判定报文段发生差错。

  • 差错处理策略

    • 检测到错误的报文段将被直接丢弃，不向上交付；若接收方采用累计确认机制，则不会为该报文段生成新的确认号。
    • 发送方依靠“超时重传 / 冗余 ACK 触发重传”机制重新发送丢弃的报文段，实现端到端可靠性。

  • 考试提示

    • 与 IP 首部检验和仅覆盖首部不同，TCP 检验和覆盖首部 + 数据，是保证端到端可靠的第一道防线。
    • TCP 检验和为必需字段；IPv4 中 UDP 检验和理论上可为 0（不使用），但 TCP 不允许省略，这一差异常被命题比较。

TCP连接管理

  • 三次握手（建立连接）

    • 定义：TCP通过三次交互确认双方的序列号和连接意愿，建立可靠连接的过程。
    • 算法步骤（客户端主动发起，初始序号：客户端初始ISN=x，服务器初始ISN=y）：
      1. 客户端→服务器：发送SYN报文段（SYN=1，seq=x），不携带数据，消耗序号x；客户端进入SYN-SENT状态。
      2. 服务器→客户端：发送SYN+ACK报文段（SYN=1，ACK=1，seq=y，ack=x+1），不携带数据，消耗序号y；服务器进入SYN-RCVD状态。
      3. 客户端→服务器：发送ACK报文段（ACK=1，seq=x+1，ack=y+1），可携带数据（不携带则不消耗序号）；客户端进入ESTABLISHED状态，服务器收到后也进入ESTABLISHED状态，连接建立完成。
      • 补充说明：第三次ACK若无数据，其序号固定为“客户端初始ISN + 1”（即x+1），因客户端仅发送过1个SYN报文段（消耗序号x），无其他数据，序号不再额外递增。
    • 核心目的：
      • 确保双方都能接收对方的信息（解决“确认接收能力”问题）。
      • 协商初始序列号（避免因序号重复导致数据混淆）。

  • 四次挥手（释放连接）

    • 定义：TCP通过四次交互确保双方数据均已传输完毕，安全释放连接的过程（支持任意一方主动关闭）。
    • 算法步骤（客户端主动关闭，统一变量：客户端已发数据最后序号=u，服务器已发数据最后序号=w）：
      1. 客户端→服务器：发送FIN报文段（FIN=1，seq=u），表示客户端不再发送数据；客户端进入FIN-WAIT-1状态（FIN报文段消耗序号u）。
      2. 服务器→客户端：发送ACK报文段（ACK=1，seq=w，ack=u+1），确认收到FIN；服务器进入CLOSE-WAIT状态（服务器无新数据发送，序号仍为w），客户端收到后进入FIN-WAIT-2状态（连接半关闭，服务器仍可发送数据）。
      3. 服务器→客户端：若服务器无数据发送，发送FIN+ACK报文段（FIN=1，ACK=1，seq=w，ack=u+1）；服务器进入LAST-ACK状态（服务器无新数据，序号仍为w，FIN报文段消耗序号w）。
      4. 客户端→服务器：发送ACK报文段（ACK=1，seq=u+1，ack=w+1）；客户端进入TIME-WAIT状态，服务器收到后进入CLOSED状态。
    • 关键状态说明：
      • TIME-WAIT：客户端需等待2MSL（最长报文段寿命）后才进入CLOSED状态，核心作用有二：① 确保服务器收到最终ACK（若ACK丢失，服务器会重发FIN，客户端可在2MSL内再次确认）；② 等待网络中残留的旧报文段过期（避免旧报文干扰新连接）。
      • CLOSE-WAIT：服务器处于“被动关闭”状态，需等待应用层处理完剩余数据后再发送FIN。

  • TCP保活计时器机制

    • 定义：TCP为避免客户机突然故障（如崩溃、断网）导致服务器长期无效等待，而设置的连接维护计时器，用于探测客户机的存活状态。
    • 核心作用：检测异常断开的连接，及时释放服务器资源（如端口、缓冲区），避免资源浪费。
    • 工作流程：
      1. 服务器每收到一次客户机发送的数据，就重置保活计时器（计时器初始值由系统配置）。
      2. 若保活计时器到期后仍未收到客户机的数据，服务器每隔75秒向客户机发送一个“探测报文段”（无数据，仅用于探测）。
      3. 若连续发送10个探测报文段后仍未收到客户机的响应，服务器判定客户机已故障，关闭该TCP连接。

TCP可靠传输机制

  • 序号与确认号机制

    • 定义：通过序号标识每个字节的数据，通过确认号告知对方已接收的数据范围，确保数据按序交付。
    • 核心规则：
      • 序号：每个TCP报文段的序号为其数据部分第一个字节的序号，连接建立时协商初始序号（ISN）。
      • 确认号：若确认号为N，表示到序号N-1为止的所有字节已正确接收，期望下一个字节序号为N。
      • 累积确认：仅确认到第一个未丢失字节的序号，无需逐个确认，减少开销（例如：收到序号301~400、501~600，确认号为401，表明400及之前已接收）。

  • 超时重传机制

    • 定义：发送方为每个报文段设置超时计时器，若计时器到期未收到确认，则重传该报文段，确保数据不丢失。
    • 超时时间（RTO）：基于往返时延（RTT）动态调整（非固定值），避免因网络时延波动导致误重传（例如：RTT增大时，RTO同步增大，减少不必要的重传）。

  • 冗余ACK（快速重传）机制

    • 定义：当接收方收到失序报文段时，连续发送3个相同的确认号（冗余ACK，不计算第一个正确的必须的ack），触发发送方立即重传失序报文段，无需等待超时；不同TCP实现（Tahoe/Reno）处理逻辑存在差异（考研核心考简化Reno）。
    • 算法步骤：
      1. 接收方收到序号为100的报文段（期望100），后收到150、200、250（失序），连续发送3个确认号为100的ACK。
      2. 发送方收到3个冗余ACK，根据不同实现处理：
      • Tahoe版本：无快恢复机制，直接将慢开始门限（ssthresh）设为当前拥塞窗口（cwnd）的一半，cwnd重置为1 MSS，重新执行慢开始算法。
      • 简化Reno版本（考研核心）：① ssthresh = 当前cwnd的一半；② cwnd = ssthresh，立即重传失序报文段；③ 进入快恢复阶段，每收到1个非冗余ACK，cwnd加1 MSS；④ 收到确认失序报文段的新ACK后，cwnd = ssthresh，切换为拥塞避免算法。

TCP流量控制与拥塞控制的对比及实现

  • 流量控制（基于滑动窗口）

    • 定义：通过接收方的接收窗口（rwnd）限制发送方的发送速率，防止发送方发送过快导致接收方缓冲区溢出，是“端到端”的控制机制。
    • 核心机制：
      • 接收窗口（rwnd）：接收方通过TCP首部的“窗口”字段告知发送方当前可用的缓冲区大小（rwnd = 接收缓冲区总大小 - 已接收未交付数据大小）。
      • 发送窗口：发送方的实际发送窗口大小 = min(rwnd, cwnd)（cwnd为拥塞窗口，由拥塞控制决定）。
      • 持续计时器：若发送方收到“零窗口”通知（rwnd=0），启动持续计时器；计时器超时后发送“零窗口探测报文段”（1字节或空ACK），查询接收方最新窗口大小，避免因窗口更新丢失导致发送方长期阻塞。

  • 拥塞控制

    • 定义：通过调整发送方的拥塞窗口（cwnd）控制注入网络的数据量，防止过多数据导致路由器/链路过载，避免网络拥塞，是“全局性”的控制机制（涉及全网设备）。
    • 核心机制：发送窗口 = min(rwnd, cwnd)，cwnd由“慢开始、拥塞避免、快重传、快恢复”四大算法动态调整。
    • 核心算法流程：
      1. 慢开始：连接建立初期/超时后，cwnd从1 MSS开始，每经过1个RTT加倍（指数增长），直至cwnd达到慢开始门限（ssthresh），切换为拥塞避免。
      2. 拥塞避免：cwnd ≥ ssthresh时，每经过1个RTT，cwnd加1 MSS（线性增长），缓慢增加发送量，避免触发拥塞。
      3. 快重传：收到3个冗余ACK时，触发立即重传，不等待超时。
      4. 快恢复：仅Reno及后续版本支持，触发快重传后，cwnd从ssthresh开始线性增长，避免重新慢开始导致的资源浪费。

  • 流量控制与拥塞控制的对比

    • 对比维度 流量控制（Flow Control） 拥塞控制（Congestion Control）
      控制对象 端到端（发送方进程 ↔ 接收方进程） 全局性（发送方 ↔ 全网路由器/链路）
      控制依据 接收方缓冲区大小（rwnd） 网络拥塞状态（超时、冗余ACK）
      控制窗口 接收窗口（rwnd） 拥塞窗口（cwnd）
      核心目标 防止接收方缓冲区溢出，确保接收方来得及处理 防止网络过载，避免路由器丢包、时延增大
      控制方 接收方主导（通过rwnd告知发送方） 发送方主导（通过网络反馈调整cwnd）
      适用场景 仅关注通信双方的接收能力 关注整个网络的承载能力
